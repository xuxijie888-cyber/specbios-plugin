# 完整认知档案（SpecBIOS Claude Code Plugin）

## 0. 范围与证据等级
- **项目类型**: Claude Code 插件开发
- **目标用户**: 使用 Claude Code 进行长期项目开发的程序员
- **证据来源**: 
  - 用户反馈：CLI 工具单独使用体验不够友好
  - 市场调研：Claude Code 有完整的插件系统
  - 技术验证：插件结构简单（Markdown + JSON）

## 1. 项目来源
- **背景**: SpecBIOS CLI 工具（v1.0.0）已发布到 npm 和 GitHub
- **痛点**: 用户需要在终端运行 `specbios dispatch`，然后复制输出到 Claude Code，体验割裂
- **机会**: Claude Code 支持插件系统，可以将 SpecBIOS 集成为原生命令
- **决策时间**: 2026-04-06，在 SpecBIOS v1.0.0 发布后立即启动

## 2. 要解决的根问题
### 核心问题
**SpecBIOS CLI 与 Claude Code 的集成体验不够流畅**

### 具体表现
1. 需要在两个工具之间切换（终端 ↔ Claude Code）
2. 需要手动复制粘贴调度输出
3. 无法自动检测 SpecBIOS 项目
4. 用户需要记住 CLI 命令语法

### 理想状态
1. 在 Claude Code 中直接使用 `/specbios-*` 命令
2. 打开 SpecBIOS 项目时自动提示当前任务
3. 一键调度和执行任务
4. 无缝的工作流体验

## 3. 核心机会
### 技术机会
- Claude Code 插件系统成熟（2026 年初已完善）
- 插件开发简单：Markdown + JSON，无需复杂构建
- 可以利用 Claude Code 的所有能力（读文件、写代码、执行命令）

### 市场机会
- Claude Code 用户量大且活跃
- 文档驱动开发理念契合 Claude Code 的工作方式
- 目前没有类似的项目管理插件

### 用户价值
- **10倍体验提升**: 从"切换工具+复制粘贴"到"一个命令"
- **自动化**: 项目检测、上下文加载全自动
- **降低门槛**: 不需要学习 CLI，直接用斜杠命令

## 4. 参照对象
### 同类插件
- **everything-claude-code**: 50k stars 的插件集合，证明插件生态活跃
- **各种 MCP 插件**: 展示了 Claude Code 的扩展能力

### 对标产品
- **Aider**: 有 `/architect` 模式，但没有持久化文档系统
- **Cursor**: IDE 集成好，但缺乏独立的任务编排层

### 差异化
- **文档驱动**: 唯一将文档作为权威上下文源的插件
- **任务编排**: 不只是代码编辑，而是完整的项目管理
- **执行器无关**: 插件本身不绑定特定 AI，可配合 CLI 使用

## 5. 当前最重要判断
### 关键假设
1. **用户会写文档**: SpecBIOS 要求用户维护项目文档，这是门槛
   - **验证方式**: 发布后观察用户反馈和使用数据
   
2. **插件比 CLI 更好用**: 集成到 Claude Code 确实能提升体验
   - **验证方式**: 对比 CLI 和插件的使用频率
   
3. **文档驱动优于聊天驱动**: 长期项目需要持久化上下文
   - **验证方式**: 跟踪用户在多天/多周项目中的使用情况

### MVP 范围决策
**做什么**:
- 5 个核心命令（init, task-add, task-list, task-update, dispatch）
- 自动检测钩子
- 基础文档

**不做什么**（v1.0.0）:
- 依赖分析（CLI 已有，插件暂不重复）
- 执行器推荐（CLI 已有）
- GUI 界面（留给未来的 Web 版本）
- 团队协作功能

### 发布策略
1. **快速验证**: 1-2 周内发布 MVP，收集反馈
2. **GitHub 优先**: 先发布到 GitHub，方便迭代
3. **插件市场**: 功能稳定后提交到官方市场
4. **文档先行**: 写好使用指南，降低学习成本

### 成功指标
- **短期（1 个月）**: 10+ 真实用户安装使用
- **中期（3 个月）**: 用户反馈插件比 CLI 好用
- **长期（6 个月）**: 成为 Claude Code 项目管理的标准插件
