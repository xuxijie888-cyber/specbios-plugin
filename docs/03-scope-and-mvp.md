# 范围与 MVP 定义（SpecBIOS Claude Code Plugin）

## 1. 当前目标

### 主目标
**发布 SpecBIOS Claude Code Plugin v1.0.0，验证插件方式比 CLI 更好用**

### 具体目标
1. 用户可以在 Claude Code 中直接使用 SpecBIOS 功能
2. 体验比"CLI + 复制粘贴"流畅 10 倍
3. 1-2 周内完成开发和发布
4. 获得 10+ 真实用户反馈

### 成功标准
- ✅ 插件可以成功安装
- ✅ 5 个核心命令都能正常工作
- ✅ 自动检测功能正常
- ✅ 用户反馈体验比 CLI 好

## 2. MVP 必做能力

### 2.1 核心命令（5 个）

#### `/specbios-init`
**功能**: 初始化 SpecBIOS 项目
**必须实现**:
- 创建 `.specbios.json` 配置文件
- 创建 `docs/` 目录
- 创建所有文档模板（00-05）
- 提示用户下一步操作

**不做**:
- 交互式配置（使用默认值）
- 模板选择（只用 personal-default）
- 自定义目录结构

#### `/specbios-task-add "描述"`
**功能**: 添加新任务到任务板
**必须实现**:
- 读取现有任务板
- 生成新任务 ID（自动递增）
- 插入任务到机器可解析区
- 保存更新后的任务板

**不做**:
- 任务优先级设置
- 任务标签/分类
- 任务依赖关系（留给 CLI）

#### `/specbios-task-list`
**功能**: 列出所有任务
**必须实现**:
- 读取任务板
- 解析所有任务
- 格式化显示（状态图标 + ID + 描述）
- 显示统计（完成/进行中/待办）

**不做**:
- 任务过滤
- 任务排序
- 任务搜索

#### `/specbios-task-update T-XXX status`
**功能**: 更新任务状态
**必须实现**:
- 解析任务 ID 和新状态
- 验证状态有效性（pending/in_progress/completed）
- 更新任务板
- 警告：如果设置 in_progress 时已有其他任务 in_progress

**不做**:
- 批量更新
- 任务描述修改
- 任务删除

#### `/specbios-dispatch [T-XXX]`
**功能**: 调度并执行任务（核心功能）
**必须实现**:
- 读取项目配置
- 找到当前 in_progress 任务（或指定任务）
- 读取所有上下文文档（00, 01, 03）
- 理解任务需求
- 执行任务（写代码/创建文件）
- 更新任务状态为 completed
- 提示继续下一个任务

**不做**:
- 任务拆分
- 依赖分析
- 执行器选择（只用 Claude Code）
- 回滚功能

### 2.2 自动检测（1 个 Hook）

#### `on-start.md`
**功能**: 会话启动时自动检测 SpecBIOS 项目
**必须实现**:
- 检查 `.specbios.json` 是否存在
- 读取项目名称
- 读取任务板，显示当前状态
- 建议下一步操作

**不做**:
- 自动执行任务
- 定时检查
- 多项目管理

### 2.3 文档

#### README.md
**必须包含**:
- 项目简介（解决什么问题）
- 安装方法
- 快速开始（5 步）
- 命令列表
- 示例工作流

#### GUIDE.md
**必须包含**:
- 中英文双语
- 详细使用说明
- 常见问题

#### LICENSE
**必须包含**:
- MIT 许可证

## 4. MVP 暂不纳入

### 4.1 高级功能（留给 CLI）

#### 依赖分析
- **原因**: CLI 已实现，插件暂不重复
- **未来**: v1.3 可以调用 CLI 的依赖分析

#### 执行器推荐
- **原因**: 插件只用 Claude Code，不需要推荐
- **未来**: 如果支持多执行器，再添加

#### AI 任务拆分
- **原因**: 需要调用外部 API，增加复杂度
- **未来**: v2.0 通过 MCP Server 集成

### 4.2 团队协作功能

#### 多人任务分配
- **原因**: MVP 专注个人使用
- **未来**: v2.0 团队版

#### 任务评论/讨论
- **原因**: 超出 MVP 范围
- **未来**: v2.0 团队版

#### 进度同步
- **原因**: 需要后端服务
- **未来**: Web 版本

### 4.3 GUI 界面

#### 可视化任务板
- **原因**: 插件是命令行形式
- **未来**: 独立的 Web/Desktop 应用

#### 依赖关系图
- **原因**: 需要图形渲染
- **未来**: Web 版本

#### 进度仪表盘
- **原因**: 超出插件能力
- **未来**: Web 版本

### 4.4 集成功能

#### Git Hooks 集成
- **原因**: CLI 已实现
- **未来**: v1.4 可以在插件中触发

#### CI/CD 集成
- **原因**: 超出 MVP 范围
- **未来**: v2.0

#### 第三方工具集成（Jira/Linear）
- **原因**: 需要 API 集成，复杂度高
- **未来**: v2.0 通过 MCP Server

### 4.5 高级配置

#### 自定义工作流
- **原因**: 增加复杂度
- **未来**: v1.5

#### 插件系统（插件的插件）
- **原因**: 过度设计
- **未来**: 根据需求决定

#### 多语言支持（i18n）
- **原因**: 文档已支持中英文，命令暂不需要
- **未来**: v1.3

## 5. 边界与约束

### 5.1 功能边界

**做什么**:
- 项目初始化
- 任务管理（增删改查）
- 任务调度与执行
- 自动检测

**不做什么**:
- 代码生成（由 Claude 完成）
- 测试执行（由 Claude 完成）
- 版本控制（由 git 完成）
- 依赖管理（由包管理器完成）

### 5.2 技术边界

**依赖**:
- Claude Code 插件系统
- Claude 的文件读写能力
- Markdown + JSON

**不依赖**:
- 外部 API
- 数据库
- 构建工具
- 第三方库

### 5.3 用户边界

**目标用户**:
- 使用 Claude Code 的程序员
- 做长期项目（多天/多周）
- 愿意写文档

**非目标用户**:
- 不用 Claude Code 的用户（用 CLI）
- 只做一次性任务的用户
- 不愿意写文档的用户

## 7. 不做的理由总结

| 功能 | 不做的理由 | 何时做 |
|------|-----------|--------|
| 依赖分析 | CLI 已有 | v1.3 集成 |
| 执行器推荐 | 插件只用 Claude Code | v2.0 多执行器支持时 |
| 任务拆分 | 需要外部 API | v2.0 MCP 集成 |
| 团队协作 | 超出个人使用范围 | v2.0 团队版 |
| GUI 界面 | 插件是命令行形式 | 独立应用 |
| Git Hooks | CLI 已有 | v1.4 |
| CI/CD | 超出范围 | v2.0 |
| 自定义工作流 | 增加复杂度 | v1.5 根据需求 |
| 多语言 i18n | 文档已双语 | v1.3 |

**核心原则**: 保持简单，快速验证，根据反馈迭代。
